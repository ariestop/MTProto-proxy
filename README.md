# MTProxy — скрипт запуска (Fake TLS)

Интерактивный Bash-скрипт поднимает [MTProto Proxy](https://github.com/TelegramMessenger/MTProxy) в Docker с режимом **Fake TLS**: генерирует секреты, подсказывает шаги для [@MTProxybot](https://t.me/MTProxybot), запускает контейнер и сохраняет параметры в файл конфигурации. После запуска открывается **меню** (перезапуск, остановка, удаление, данные подключения, статус, логи).

## Требования

- **ОС:** Linux (рекомендуется для прокси-сервера) или macOS; на сервере обычно доступен `ss`, локально на macOS может использоваться `lsof`.
- **Docker** с правами на запуск контейнеров (скрипт вызывает `sudo docker`).
- Утилиты в `PATH`: `bash`, `curl`, `xxd`, `openssl`, `grep`; для проверки портов — **`ss`** или **`lsof`** (хотя бы одна).

Установка Docker и добавление пользователя в группу `docker` — по документации вашей системы. Без `sudo` для `docker` убедитесь, что текущий пользователь может выполнять `docker` без пароля, либо будьте готовы вводить пароль при каждом вызове `sudo`.

## Установка

1. Склонируйте или скопируйте каталог проекта на сервер.
2. Сделайте скрипт исполняемым:

```bash
chmod +x start-mtproxy.sh
```

Дополнительных зависимостей (Python, Node и т.д.) нет.

### Сборка своего образа из исходников GitHub (опционально)

По умолчанию **`start-mtproxy.sh`** тянет **`telegrammessenger/proxy:latest`** с Docker Hub. Если нужен бинарник с ветки `master` (другая версия `mtproto-proxy`, см. [README MTProxy](https://github.com/TelegramMessenger/MTProxy/blob/master/README.md)), соберите локальный образ:

```bash
chmod +x install-mtproxy.sh
./install-mtproxy.sh
```

Опции:

- `-t образ:тег` или переменная `MTPROXY_IMAGE` — имя локального образа (по умолчанию `local/mtproxy:latest`).
- `--no-cache` — полная пересборка без кэша слоёв.

Скрипт выполняет `docker build` по каталогу `docker-mtproxy/`: на стадии **builder** клонируется ветка `master` с GitHub и выполняется `make`, в финальный образ попадает только `mtproto-proxy` и `entrypoint.sh`. При старте контейнера подтягиваются `proxy-secret` и `proxy-multi.conf` с `core.telegram.org` (как в документации MTProxy). Entrypoint автоматически определяет внешний IP и передаёт `--nat-info` бинарнику, чтобы прокси корректно работал за Docker NAT.

**Перенос на другой сервер:** после сборки можно согласиться сохранить архив или вручную:

```bash
docker save "local/mtproxy:latest" | gzip > mtproxy-image.tar.gz
# на целевом хосте:
gunzip -c mtproxy-image.tar.gz | docker load
```

После сборки скрипт предложит запустить `start-mtproxy.sh` с `DOCKER_IMAGE`, указывающим на собранный образ.

## Запуск

```bash
./start-mtproxy.sh
```

При первом запуске образ по умолчанию — **`telegrammessenger/proxy:latest`** (Docker Hub). Выбор **Hub** или **локальной сборки** — пункт меню **8**; выбор сохраняется в **`~/.mtproxy_docker_image`** и (если есть) обновляет `IMAGE=` в `~/mtproto_config.txt`. Переменная окружения **`DOCKER_IMAGE`** при старте скрипта имеет приоритет над этим файлом.

Откроется меню:

| Пункт | Действие |
|-------|----------|
| 1 | Установка / переустановка (если контейнер уже есть — запрос на удаление) |
| 2 | Перезапуск контейнера |
| 3 | Остановка |
| 4 | Удаление контейнера (опционально — удаление `mtproto_config.txt`) |
| 5 | Данные подключения из конфига + блок **без цветов** для копирования строк |
| 6 | Статус контейнера, конфигурация и **текущий образ Docker** |
| 7 | Последние строки логов Docker |
| 8 | Образ Docker: Hub (`telegrammessenger/proxy:latest`) или локальный (`local/mtproxy:latest`) |
| 9 | Статистика абонентов: отчёт по IP и времени сессий, запуск/остановка фонового сборщика (`stats-mtproxy.sh`) |
| 0 | Выход |

При **установке (п. 1)** скрипт последовательно запросит:

1. **Внешний IP или hostname** — для бота и ссылки `tg://proxy`; пустой ввод подставляет адрес с `https://ifconfig.me/ip` (если удалось).
2. **Порт хоста** — на `443` внутри контейнера. По умолчанию — первый свободный из `443`, `8443`, `8444`, `8445`. Если выбранный порт занят, предложат ввести свободный.
3. **Домен Fake TLS** — по умолчанию `ya.ru` (клиентский секрет).
4. Инструкции для **@MTProxybot** и ввод **TAG** (32 hex или пусто).

### Docker-образ

- **Пункт меню 8** — переключение между **`telegrammessenger/proxy:latest`** (Hub) и **`local/mtproxy:latest`** (сборка из [MTProxy](https://github.com/TelegramMessenger/MTProxy) через `./install-mtproxy.sh`).
- Без меню: однократно задать образ при запуске:

```bash
DOCKER_IMAGE=local/mtproxy:latest ./start-mtproxy.sh
```

Свой реестр (приоритет у переменной окружения):

```bash
DOCKER_IMAGE=registry.example.com/mtproxy:stable ./start-mtproxy.sh
```

## Fake TLS secret (одна строка)

Скрипт генерирует **одну** строку secret в стиле рабочего сценария Fake TLS:

`ee` + **hex(домен)** + случайный hex так, чтобы после префикса `ee` было **ровно 30** шестнадцатеричных символов (длина домена в hex не больше 30).

Эту **же** строку нужно передать в:

- `docker … -e SECRET=…`
- поле Secret в [@MTProxybot](https://t.me/MTProxybot)
- параметр `secret=` в `tg://proxy?…` и в приложении Telegram

Это **не** описано в кратком [README MTProxy](https://github.com/TelegramMessenger/MTProxy/blob/master/README.md) (там обычный hex-секрет и опциональный префикс `dd` для padding на клиенте).

## Файл конфигурации

После успешного старта создаётся **`~/mtproto_config.txt`** с правами **600**:

- `SERVER`, `PORT`, `SECRET`, `DOMAIN`, `TAG`, `LINK`, `IMAGE`.

Пункт меню «Данные подключения» читает `SECRET`; для старых конфигов допускается поле `CLIENT_SECRET` вместо `SECRET`.

Не публикуйте этот файл: в нём секреты прокси. Чтение для пункта меню «Данные подключения» выполняется **без** `source` (значения не исполняются как shell-код). После установки и в пункте 5 дополнительно выводится рамка **«Скопируйте значения»**: чистый текст без ANSI на отдельных строках — удобно выделить мышью.

## Статистика абонентов (п. 9 меню)

Скрипт **`stats-mtproxy.sh`** на **Linux-хосте** с Docker подписывается на события **`conntrack`** для TCP с **портом хоста** из `PORT=` в `~/mtproto_config.txt` и накапливает длительности сессий в **`~/.mtproxy_stats/sessions.tsv`**.

- **Первое соединение** — по минимальному времени начала сессии для данного IP.
- **Время за сегодня / 7 / 30 дней / всего** — сумма длительностей с пересечением соответствующего интервала (календарный «сегодня» — по локальному времени сервера, нужен GNU `date`).
- Виден **IP** (и порт источника); за **NAT** один адрес может соответствовать многим пользователям.
- Учитываются сессии **после** запуска сборщика; при обрыве процесса без события `DESTROY` часть сессий может не попасть в файл.

**Зависимости:** пакет **conntrack-tools** (`conntrack`), загруженный **nf_conntrack**. Если `conntrack -E` требует прав root, запускайте сборщик в среде, где эти права есть (или настройте доступ к netlink — по политике вашей ОС).

```bash
chmod +x stats-mtproxy.sh
./stats-mtproxy.sh start    # фон, лог ~/.mtproxy_stats/collector.log
./stats-mtproxy.sh report   # таблица
./stats-mtproxy.sh stop
```

То же из меню **`./start-mtproxy.sh`** → п. **9**.

## Полезные команды

```bash
sudo docker ps
sudo docker logs mtproto-proxy
sudo docker restart mtproto-proxy
sudo docker stop mtproto-proxy
```

Имя контейнера: **`mtproto-proxy`**. Том **`mtproxy-data`:**`/data` монтируется для единообразия; в образе из **`install-mtproxy.sh`** данные прокси лежат в **`/var/lib/mtproxy`** внутри контейнера, `/data` может быть пустым.

## Устранение неполадок

- **«Не найдена команда»** — установите недостающую утилиту (`curl`, `xxd` из пакета `vim`/`xxd`, `openssl`).
- **Нет `ss` и нет `lsof`** — установите один из вариантов (например, `iproute2` для `ss` на Linux).
- **Порт занят** — выберите другой порт или освободите текущий; веб-сервер на `443` конфликтует с пробросом на тот же порт хоста.
- **Контейнер не стартует** — смотрите `sudo docker logs mtproto-proxy`; проверьте, что образ совместим с переменными `SECRET` / `TAG`.
- **Авто-IP неверный** — введите внешний IPv4 или hostname вручную при запросе.

### Прокси «подключён», но сообщения не грузятся (нет трафика)

MTProxy **проксирует только клиент Telegram**, не весь интернет телефона.

1. **NAT / Docker** — прокси **обязан** знать внешний IP сервера (`--nat-info`). Entrypoint локального образа делает это автоматически: берёт `EXTERNAL_IP` из env (передаётся скриптом запуска) или определяет через curl. Если вы запускаете контейнер вручную — передайте `-e EXTERNAL_IP=<ваш_публичный_IP>`.
2. **Совпадение секрета** — в Docker переменная **`SECRET`** должна **побитово совпадать** с тем, что вы отправили в @MTProxybot и что в поле **`SECRET`** (или старом `CLIENT_SECRET`) в конфиге. Один неверный символ даёт сбой проверки или трафик без данных.
3. **TAG** — в контейнере **`TAG`** должен совпадать с тем, что выдал бот для **этой** пары адрес + secret. После смены secret заново пройдите `/newproxy` и введите новый TAG при установке.
4. **Исходящий доступ с сервера** — контейнер должен достигать сети Telegram (загрузка `proxy-multi.conf` при старте уже идёт наружу). Проверка с ВМ:  
   `curl -fsS --max-time 10 -o /dev/null -w '%{http_code}\n' https://core.telegram.org/getProxyConfig`  
   Должен быть **200**. Если в облаке жёсткий egress-фильтр — разрешите исходящий HTTPS (и типичные порты к DC Telegram).
5. **Ссылка** — `tg://proxy?...&secret=` должен содержать **ту же** строку, что и `SECRET` в Docker (у этого проекта по умолчанию это строка с префиксом `ee`).
6. **`nf_conntrack` table full** — если сервер одновременно шлюз/VPN, ядро может отбрасывать пакеты при заполнении таблицы conntrack. Симптом — `nf_conntrack: table full, dropping packet` в `dmesg`. Проверка и фикс:
   ```bash
   sysctl net.netfilter.nf_conntrack_max net.netfilter.nf_conntrack_count
   echo 'net.netfilter.nf_conntrack_max=262144' | sudo tee /etc/sysctl.d/99-conntrack.conf
   sudo sysctl --system
   ```

### Проблемы с проверкой MTProto

Сбои возможны из‑за версии клиента, DPI или сети. Имеет смысл сменить Wi‑Fi / LTE, обновить Telegram или попробовать другой клиент. В [README MTProxy](https://github.com/TelegramMessenger/MTProxy/blob/master/README.md) описан префикс **`dd`** для random padding — **другой** режим относительно `ee`, на клиенте иногда помогает в «подозрительных» сетях.

При проблемах включите **`sudo docker logs -f mtproto-proxy`** и повторите открытие чата в Telegram — по новым строкам в логе проще понять, доходят ли запросы до прокси.

## Безопасность

- Секреты дают доступ к вашему прокси; храните `mtproto_config.txt` в безопасности.
- Скрипт валидирует порт, адрес сервера и формат TAG, чтобы снизить риск ошибок и неожиданного разбора аргументов у `docker`.
- Используйте актуальный и доверенный образ MTProxy и следите за обновлениями ОС и Docker.
